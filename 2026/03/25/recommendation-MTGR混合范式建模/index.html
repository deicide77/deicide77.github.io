MTGR:混合范式建模

前言

生成式推荐的用户粒度建模范式，与传统DLRM的特征工程经验相结合：在保持用户粒度聚合效率的同时，支持target-aware的判别式建模，

范式的再思考：生成 vs 判别

HSTU和GenRank基于内容和行为交替的序列建模；但是他们仍然保留的限制是：无法将交叉特征（历史统计）以不打破序列建模的因果性的前提下，融入到自回归方式中，去实现生成式推荐。

在目前的生成式推荐过程中，仍然需要交叉特征去提取用户行为信息，或者是特征相关性；这是经过美团的消融验证：去除交叉特征后，即使是最大规模的生成式模型，性能也会退化到甚至不如中等规模的传统DLRM。

判别式排序的本质

首先在判别式推荐中，交叉特征的作用就像：先拿到候选的某个属性，再回头去用户历史里筛选相关子集，然后基于这个子集计算统计量，即“在满足候选属性条件的历史子集上，做统计，再用于当前预测。”这就是在不同的条件聚合下去做判别搜索；

但是生成式模型不一样：
生成式模型通常把任务写成：

给定历史序列
预测下一个 item，或者生成一个 item token / semantic ID

因为交叉特征不是单纯的“理解用户历史”，而是“在候选条件下重解释用户历史”；而生成式建模更擅长整体建模和分布预测，所以不擅长显式地做这种候选驱动的条件筛选与统计聚合。

这当中其实最凸显的问题就是：生成式对于一个用户的预测通常只有一个总的用户状态，它如果不利用条件聚合，生成的答案不能随“时间”、“领域”随时切换视角回答；

这即是：用户粒度建模带来的效率提升，本质上来自于样本聚合和计算复用，而不一定要求完整的生成式建模。

作者提出的那句话实际上也是告诉我们一个很浅显的道理：排序本来是一个“只需要回答条件问题”的任务，但生成式方法为了得到这个条件问题的答案，被迫先去学一个更大的问题，或者说回答了更大的问题，忽略了问题本身的范围，因此多做了很多并非任务本身必需的建模。

既然整体生成不好表达局部条件交互，那是不是要把它拆开、分阶段、把局部信息重新塞回来？看完第一段我自然会有这样的想法。

MTGR的核心洞察

MTGR的核心思想我的理解是：不再把候选当作要生成出来的序列后续部分，而是把候选直接当作输入 token 放进来，然后只对这些候选 token 做判别。

读到这里，我会发现自己前面的困惑其实正在被慢慢解开。最开始我之所以会觉得生成式方法在排序任务里有些“别扭”，是因为我一直隐约感觉到，排序真正关心的并不是“下一个物品如何被生成出来”，而是“在一组已经给定的候选里，当前这个用户更可能对哪个产生行为”

如果从这个角度看，排序本质上是一个条件判别问题，它只需要学习在给定用户历史和候选物品的条件下，用户行为发生的概率，也就是更接近 (P(y \mid h, i)) 这样的目标；但生成式方法往往要绕一个更大的圈子，它需要去建模更完整的联合关系，或者至少去建模“历史如何导向下一个物品”这样的生成过程。这样做当然有它的好处，比如可以利用统一的序列建模框架、可以自然地做用户粒度聚合、可以在多个目标之间共享表示，但问题在于，它也因此承担了很多并不是排序任务本身真正需要的建模负担。尤其是在推荐排序中，很多真正强有力的信号并不来自于“整体序列生成得是否合理”，而是来自于一些很明确的、以候选为条件的局部交互信息，比如用户在当前时段对这类内容的历史点击率、用户在某个地理位置对某类商家的偏好、某个创作者的内容对某类用户群体的吸引力等等。

这些信号本质上都不是简单的用户兴趣总结，而是“在候选某个属性条件下，对用户历史的重新筛选和统计”，也就是说，候选一旦变化，需要被激活、被观察、被统计的那部分用户历史也会随之变化。这种信息如果放在判别式框架里，其实是很自然的，因为候选本来就是输入的一部分，我们完全可以围绕候选去构造交叉特征，让模型直接学习用户—候选之间的条件关系；

但如果放在纯生成式框架里，候选往往被视为未来要预测、要生成的对象，而不是当前已知、可以参与特征构造的输入对象，这就导致这些交叉信息很难被直接、自然地表达出来，它们只能被迫隐式地压缩进统一的历史表示或者参数里，让模型自己去“猜”哪些局部关系重要。

也正是在这里，我开始理解为什么作者会强调，用户粒度建模带来的效率优势，并不必然要求完整的生成式建模。过去很容易把这两件事绑定在一起：好像只要说到用户粒度聚合，就默认是在做生成式推荐；但实际上，用户粒度聚合真正带来的核心收益，更多来自于样本组织方式的改变和计算复用，而不是来自生成式目标本身.

传统 point-wise 排序中，同一个用户面对多个候选时，往往会被重复构造成许多条样本，用户的历史部分也会被反复编码很多次，这会产生大量重复计算；而如果以用户为单位组织样本，把多个候选一起放进同一个样本中，那么用户侧的历史和上下文信息其实只需要编码一次，不同候选共享同一份用户背景，这样自然就能获得效率上的显著提升。

也就是说，效率提升的本质是“共享”而不是“生成”。理解了这一点之后，MTGR 的设计就显得非常巧妙了。它真正做的并不是把原有的生成过程拆成更复杂的多阶段生成，也不是在生成框架上强行补几个局部模块，而是更根本地重新定义了候选在排序任务中的角色：候选不再被当作序列未来的一部分，不再是要生成出来的中间状态，而是被直接当作当前输入的一部分。

这样一来，整个数据组织方式就发生了本质变化。用户的静态属性、长期历史、短期实时行为仍然保留下来，继续构成用户上下文；但多个候选也被一起放进输入序列中，每个候选对应一个 token，而这个 candidate token 里不仅包含物品本身的特征，还可以直接融合那些依赖于用户历史和候选属性交互得到的交叉特征。

这样做非常关键，因为它等于承认了排序阶段的真实任务设定：候选是给定的输入，而不是等待生成的输出。既然候选已经在输入里，我们就没有必要再遵守“内容—行为交替”的严格时序约束，也没有必要强迫候选去伪装成自然行为序列的下一项。我们可以更坦率地让模型看到“当前有哪些候选”“每个候选和用户之间有哪些条件化关系”，然后只让模型去回答一个排序问题：这些候选里哪个更值得被点击、被停留、被转化。从这个角度看，MTGR 最有意思的地方就在于它**区分了“架构”和“目标”**这两个常常被混在一起的东西。

Transformer 这种序列模型是一种非常强的表征学习工具，它擅长在统一的 token 空间中做上下文建模和关系交互；用户粒度聚合是一种很高效的样本组织方式，它能够让用户侧信息被一次编码、多候选共享；但这两者并不天然要求我们一定去做生成式建模。也就是说，生成式架构并不等于生成式目标，用户粒度聚合也不等于必须建模完整的行为序列。MTGR 正是把这两层拆开了：它保留了 Transformer 和用户粒度聚合，因为它们在表征能力和计算效率上都非常有价值；但它放弃了完整生成式目标，转而只在候选 token 上计算 loss，只预测用户对这些已知候选的行为。

这样一来，整个模型既保留了用户级别共享计算的效率优势，也重新获得了判别式建模在交叉特征表达上的灵活性。我会觉得这种设计特别值得注意，因为它不是简单地在“生成式”与“判别式”之间二选一，而是更深入地回答了一个问题：我们真正想保留的到底是什么？

如果效率来源于用户粒度的计算复用，那我们就保留这种组织方式；如果排序本质要求候选条件下的局部交互表达，那我们就回到判别式目标；如果 Transformer 的优势在于统一上下文建模，那我们就继续使用它作为表示学习手段。

这样看，MTGR 并不是一个折中得模糊的方案，反而是一个边界划分得很清楚的方案：它把该保留的手段保留下来，把不再适合任务本质的目标拿掉，最终形成了一种“用生成式架构服务判别式排序”的混合范式。

对我来说，这一段最重要的启发在于，它提醒我在看模型设计时不能只看表面形式，比如“是不是 Transformer”“是不是用户粒度”“是不是序列建模”，而要继续追问：这个架构到底是在服务什么目标，这个目标是不是和任务本质一致，哪些部分是真正必要的，哪些部分只是某种历史路径依赖下被默认绑在一起的选择。

MTGR 的价值恰恰就在于，它把这些原本被绑在一起的东西重新拆开，并且证明了：排序任务完全可以在保留用户粒度高效性的同时，不必承担完整生成式建模的代价。

架构创新的原理

特征到Token的映射

HSTU 和 GenRank 的启发，是它们确实把传统 DLRM 里原本分散在不同模块中的异构特征，尽可能拉到了同一个表征框架中。

也正因如此，HSTU 和 GenRank 试图做一件很有吸引力的事情，那就是把所有信息统一编码成 token 序列，用一个共享的 Transformer 去处理。这样一来，原来分散在各个模块里的信息都被投影到一个统一的 token 化框架中，从工程和建模视角看都很优雅，因为它意味着我们不再需要为不同类型特征维护一套完全不同的计算路径，而是可以借助 Transformer 的共享参数和自注意力机制，在一个统一空间中建模各种上下文关系。

但当我继续往下看，尤其是把交叉特征也考虑进来之后，“统一”并不等于“同质”。也就是说，形式上的统一 token 化，并不意味着这些 token 在语义上属于同一种空间。这个问题在 MTGR 里会更加明显，因为候选 token 不再只是简单的 item token，而是融合了物品固有特征、用户—候选交叉特征以及位置/曝光等上下文信息的复合表示。对于第 (j) 个候选，MTGR 实际上是在构造这样一个表示：

[
c_j = f_{\text{cand}}(x^{\text{item}}_j,; x^{\text{cross}}_j,; x^{\text{ctx}}_j),
]

其中，(x^{\text{item}}_j) 表示候选自身的物品特征，例如 ID、类目、标签、时长等；(x^{\text{cross}}_j) 表示与该候选绑定的交叉特征，例如用户对该类目的历史点击率、用户在当前时段对该类内容的偏好等；(x^{\text{ctx}}j) 表示位置和时序上下文，例如曝光位置、曝光时间等。更进一步，如果写得更细一点，可以把它表示为：
[
c_j = \phi{\text{cand}}!\left(
e^{\text{id}}_j,;
e^{\text{cat}}_j,;
e^{\text{tag}}_j,;
r^{\text{cross}}_j,;
p^{\text{pos}}_j,;
t^{\text{time}}j
\right),
]
其中 (\phi{\text{cand}}(\cdot)) 可以理解成 embedding、MLP 和特征融合后的候选 token 构造函数，(e^{\text{id}}_j, e^{\text{cat}}_j, e^{\text{tag}}_j) 分别对应物品 ID、类目、标签等离散特征的嵌入，(r^{\text{cross}}_j) 表示连续或统计型交叉特征，(p^{\text{pos}}_j) 表示位置编码，(t^{\text{time}}_j) 表示时间相关特征。这里我觉得最关键的一点是，交叉特征被明确视为候选表示的一部分，而不是历史序列的一部分。因为交叉特征本质上是“针对当前候选、在当前用户条件下”计算出来的，它天然是备选。比如对于三个候选，候选 1 和候选 3 都是科技类内容，同一用户对科技类内容的历史点击率都是 (0.8)，于是它们的某部分交叉特征会相同；但因为它们是两个不同的候选，在其他物品属性上仍可能不同，所以依然应该被表示成两个独立 token，而不是共享同一个“科技类 CTR token”。这说明候选 token 的本质不是简单的“某个交叉统计量”，而是一个围绕具体候选构造的复合语义单元。

相对而言，用户历史部分的 token 构造要“纯粹”得多。用户属性 token 可以写作
[
u_m = f_{\text{user}}(x^{\text{user}}m),
]
其中 (x^{\text{user}}m) 表示第 (m) 个用户静态属性，例如年龄、性别、城市等；历史序列 token 可以写作
[
s_t = f{\text{seq}}(x^{\text{seq}}t),
]
其中 (x^{\text{seq}}t) 表示第 (t) 个历史交互物品；实时行为 token 可以写作
[
r\tau = f{\text{rt}}(x^{\text{rt}}\tau),
]
其中 (x^{\text{rt}}_\tau) 表示第 (\tau) 个近期交互行为。把这些拼接起来，整个输入序列可以表示为
[
X = [u_1,\dots,u_M,; s_1,\dots,s_T,; r_1,\dots,r_R,; c_1,\dots,c_K].
]
接着，MTGR 仍然会用统一的 Transformer 去处理这个序列：
[
H = \text{Transformer}(X),
]
并且只在候选位置取出最终表示：
[
h^{\text{cand}}_j = H[M + T + R + j],
]
再通过判别头输出排序分数：
[
\hat{y}_j = g(h^{\text{cand}}j),
]
其中 (g(\cdot)) 通常可以是一个 MLP，最终输出点击率、转化率或者其他目标行为的 logit / probability。训练目标也只定义在候选位置上，例如可以写成
[
\mathcal{L} = \sum{j=1}^{K} \ell(\hat{y}_j, y_j),
]
其中 (\ell(\cdot,\cdot)) 是二分类交叉熵或其他排序损失。

真正让我仔细思考的，是由作者提出的语义不对称性。虽然所有信息最后都变成了 token，表面上看进入了同一个 Transformer，但这些 token 的含义显然并不一样。用户属性 token 编码的是人口统计学语义；序列 token 编码的是用户长期或短期行为模式，它本质上是在描述“用户过去做过什么”；而候选 token 编码的则不是单纯的物品，而是“候选物品 + 用户相关交叉关系 + 曝光上下文”的组合语义。也就是说，从信息来源上看，用户 token 更偏静态画像空间，序列 token 更偏行为动态空间，候选 token 则已经进入了目标判别空间。

它们虽然都被写成向量，都可以放进同一个注意力机制里，但这并不意味着它们原本就属于完全同质的语义空间。恰恰相反，它们更像是几个彼此相关、但语义角色不同的子空间。如果直接把这些 token 不加区分地送进一个完全统一的 self-attention，那么理论上每个 token 都可以无差别地和其他 token 交互，这就会带来一个潜在问题：不同类型 token 之间的交互未必总是有益的，甚至可能出现语义干扰。形式上，标准 self-attention 写作
[
\text{Attn}(Q,K,V)=\text{softmax}!\left(\frac{QK^\top}{\sqrt{d}}\right)V,
]
如果对整个序列统一计算，那么某个用户属性 token、某个历史序列 token、某个候选 token 之间都会被纳入同一个注意力图中。问题在于，这种“完全自由”的交互默认假设所有 token 的语义地位是平等且可直接比较的，但在这里这个假设未必成立。

比如一个表示“年龄段”的用户属性 token，和一个表示“科技类候选 + 类目 CTR = 0.8”的候选 token，它们虽然都在一个向量空间中，但它们承载的信息结构显然不一样；如果没有额外机制约束，模型可能会把不该强耦合的信号过度混合，或者在训练中让某些语义更强的 token 主导注意力，进而压制其他类型信息。

所以我会觉得，该框架的创新“统一的 token 框架下，如何保留异质信息的语义边界”。一方面继承了 HSTU 那种统一特征空间带来的参数共享和计算效率，让所有信息都能放进一个共同的 Transformer 框架中；但另一方面，它又不能假装这些信息在语义上完全同构，因为候选 token 已经明显带有判别任务特有的交叉信息，而历史 token 则仍然是 历史编码。如果不处理这种异质性，统一架构就可能从优势变成负担，因为参数虽然共享了，语义却混杂了。于是这里真正值得思考的方向是：保持不同信息仍然位于一个可交互的统一框架里，但在交互方式上引入某种结构化约束，让不同类型 token 既能共享信息，又不会完全无边界地相互污染。我们需要的不是“完全统一”或者“完全独立”这两个极端，而是一种介于二者之间的设计：既承认用户属性、行为序列、候选表示分别来自不同语义空间，又允许它们在排序任务需要的地方发生受控交互。

Group Layer Normalization的必要性

HSTU 式统一 token 化带来了参数共享和计算简化，这是它的优点；

但 MTGR 同时引入了更强的语义异质性，引入四类 token 后的功能差异比以前更明显。如果还坚持一套完全统一的归一化方式，就等于在表示层面忽略这种差异。

GLN 的作用，就是在不破坏统一 Transformer 框架的前提下，给这种差异留下空间。

它的好处可以总结成两个方面。

  • 第一是训练更稳定，因为每个 group 都可以有适合自己的缩放和平移参数，不需要为了迁就其他 token 类型而共享同一套统计调节方式；
  • 第二是语义更清晰，因为不同 group 在同一维度上可以保留相对独立的功能，不会被一套统一参数过度耦合。

这样做的好处是，模型既保留了 HSTU 那种统一处理所有 token 的工程优势，又避免了完全统一带来的语义污染问题。

从作者教程式的角度看，这一段其实很好总结：前面统一特征空间解决的是“如何让不同信息进入同一个模型”，而这里的 GLN 解决的是“进入同一个模型之后，如何避免它们被错误地当成同一种信息”。

这也是 MTGR 整个混合范式里很关键的一点：不是简单追求统一，而是在统一框架下承认差异、利用差异，并通过很小的结构调整把这种差异变成模型的优势。

Dynamic Masking：超越Causal Mask

三条规则：

  • 规则1：静态序列对所有token可见
  • 规则2：动态序列遵循因果性
  • 规则3：候选之间相互独立

通过根据每个样本中token的实际时间戳动态生成的。避免了信息泄露。推理时，它允许同一个请求中的所有候选并行处理（因为RealTime只包含请求之前的交互，候选之间相互独立），保持了计算效率。
