前言：从好奇开始

在开始正式学习推荐系统之前，我选择先阅读教程的第一章，对推荐系统整体结构进行一个宏观理解。虽然本书的核心内容从第五章开始，但作为一个刚接触推荐系统的学习者，我希望先理解推荐系统整体在解决什么问题，以及它在现代互联网系统中的角色。

在开始学习推荐系统时，我首先产生的一个好奇是：

推荐系统和搜索系统到底有什么区别？

搜索系统的逻辑相对容易理解。用户输入一个明确的查询，例如“推荐系统教程”，系统只需要在数据库中找到与这个查询最相关的内容并返回结果。搜索的本质可以概括为：

Query → Matching → Ranking

搜索系统的核心任务是根据用户明确表达的需求，在海量数据中找到最相关的信息。

然而推荐系统面对的是一个完全不同的问题。

在绝大多数情况下，用户并不会明确告诉系统自己想要什么内容。例如，当用户打开抖音、B站或者今日头条时，用户通常并不会输入任何查询。此时系统需要在没有明确指令的情况下，主动判断用户可能感兴趣的内容。即：

User → Predict Potential Interest → Recommend Items

而推荐系统需要根据用户的历史行为、兴趣偏好以及当前使用场景，主动预测用户接下来可能产生兴趣的内容。

这意味着推荐系统比搜索系统面临更复杂的挑战。搜索系统可以依赖用户提供的查询信号，而推荐系统则需要通过用户行为数据来推测用户的潜在需求。

我的学习动机

我对推荐系统产生兴趣，其实来源于两个方向。

第一个是我在进行多模态 RAG 项目时，对 检索系统（retrieval system） 有了一些基础理解。例如，在多模态检索中，CLIP 模型通过计算图像和文本的相似度来找到最相关的内容。这让我意识到，推荐系统和检索系统之间似乎存在某种联系：两者都需要在海量数据中找到最相关的内容。

第二个原因是我在阅读 生成式推荐系统相关论文 时，第一次接触到了“生成式推荐”的概念。例如高莘老师的论文 Generative Next POI Recommendation with Semantic ID。这类工作提出了一种新的思路：推荐系统不再只是对候选物品进行评分，而是像语言模型一样，通过序列建模直接生成推荐结果。

这一思路有一些对推荐系统的好奇。例如：

  • 生成式推荐和传统推荐系统之间有什么区别？
  • 推荐系统中的检索（retrieval）和排序（ranking）与 RAG 中的检索阶段是否存在共通之处？

这种挥之不去的疑惑感让我意识到，在真正学习具体算法之前，我需要先理解推荐系统整体的技术框架。

因此，我希望通过阅读本教程，从最基础的概念开始，逐步理解：

  • 推荐系统在解决什么问题
  • 推荐系统的基本架构是什么
  • 推荐算法是如何逐步发展到深度学习和生成式模型的

在阅读之前的一些疑问

在真正开始学习生成式推荐系统之前，我在阅读生成式推荐相关论文以及思考推荐系统架构时，也产生了一些疑问。这些问题可能并没有立即的答案，但我希望在后续学习过程中逐步理解它们。

1 推荐系统是否本质上是一个序列预测问题？

当我第一次看到“生成式推荐系统”时，很自然地联想到深度学习中的序列建模问题。

例如，在自然语言处理中，语言模型的核心任务是预测：

Token_t+1 = f(Token_1, Token_2, …, Token_t)

而在推荐系统中，用户的行为也天然构成一个时间序列：

Item_1 → Item_2 → Item_3 → …

如果从这个角度来看，推荐系统似乎也可以被理解为一个 序列预测问题：

Next Item = f(User Interaction History)

这是也是我的第一个疑问：

如果推荐系统天然具有序列结构，那么为什么早期推荐系统并没有采用序列模型，而是使用协同过滤等方法？

2 推荐系统是否可以完全被检索系统替代？

在进行多模态 RAG 项目时，我逐渐理解了现代检索系统的基本结构。在很多检索系统中，数据会被表示为向量（embedding），然后通过向量相似度搜索来找到最相关的内容。例如：
User Embedding → ANN Search → TopK Items
这种方式看起来与推荐系统非常相似。

因此我产生了一个疑问：

**推荐系统是否可以完全用检索系统来替代？**如果我们可以通过 embedding 表示用户兴趣和物品特征，那么似乎只需要做向量相似度搜索，就可以找到最相关的内容。但在工业推荐系统中，我们仍然看到复杂的排序模型（ranking model），例如：DeepFM、DIN、DLRM。
这让我意识到：推荐系统不仅仅是一个“找到相似内容”的问题，它可能还涉及更复杂的行为预测问题。

3 生成式推荐是否真的不需要候选集合？

生成式推荐提出了一种非常吸引人的架构：
通过一个统一的模型直接生成推荐结果，而不需要传统推荐系统中的多阶段流水线。

但在真实平台中，物品规模可能达到：1000万甚至1亿

如果使用生成式模型直接生成推荐结果，那么模型如何保证生成的是 真实存在的物品？

例如在短视频平台中：

  • 视频数量极其庞大
  • 新视频不断产生
  • 用户兴趣变化极快

在这样的场景下，我很难直观地想象一个生成式模型如何直接生成推荐内容，而不借助某种候选集合机制。

因此我也产生一个进一步的问题：

生成式推荐系统是否真的可以完全摆脱候选集合（candidate set），还是仍然需要某种形式的检索机制？

4 推荐系统究竟是在预测用户兴趣，还是在塑造用户兴趣？

最后，我想聊一个我个人认为非常重要的问题。

从技术指标上看，推荐系统的逻辑很纯粹，就是预测你的行为：

  • 点击率（CTR）
  • 观看时长
  • 购买概率

但在现实中，我观察到它并不只是在预测，而是在影响甚至塑造我们的兴趣。

这种现象已经非常明显，以前推荐系统没那么强的时候，我们就已经看到了它带来的负面效应：

  • 信息茧房：算法不断强化已知兴趣，把人困在相似的内容里。
  • 个体的原子化：虽然推送越来越精确，但结果是让人们分化成了各种小圈子。
  • 认知的固化：它打破了传统的信息壁垒，帮我们逃离了家庭或旧知识带来的创伤，但同时也带来了新的“知识封锁”。

作为一名还没正式进入工业界、正在学习推荐系统的学生，我有一种除技术之外的担忧。

我担心未来的推荐系统会从“帮助用户发现内容”，逐渐变成“决定用户看到什么内容”。尤其是如果未来演进为生成式推荐系统，这种塑造的力量会不会变得更强？

更深层的焦虑在于，如果我未来进入研发岗位，为了追求更好的算法效果，我会不会在潜意识里通过“固化用户兴趣”来达到数据上的完美？

为了让生成式推荐的效果更好、更精准，我是不是正在不自觉地参与一场“把人变简单”的过程？如果为了指标的提升，技术手段最终变成了对人兴趣的修剪和固化……

当然这些也只是一些猜测罢了 ^-^

第一章 推荐系统的最基本问题

在阅读这一章时，我首先尝试理解一个最基本的问题：推荐系统到底在解决什么问题？

教程中给了一个非常直观的例子：当我们打开一个视频应用时，系统需要在成千上万个视频中决定哪些内容最可能与你产生有价值的连接。

这里的“连接”并不是一个抽象概念，而是可以被系统观测到的具体行为。例如：

  • 用户是否点击了视频
  • 是否观看超过30秒
  • 是否点赞或者分享

这些行为在推荐系统中被称为 用户交互（interaction）。因此推荐系统的核心任务，本质上就是预测哪些内容最可能与用户产生这样的交互。

推荐系统三个对象以及两个基本范式

在看到推荐系统被拆分为 User / Item / Context 三个要素时，我一开始其实有些疑惑：为什么问题恰好会被分成这三个部分？如果只是从直觉来看，这似乎只是对信息来源的一种分类。但仔细想一想，这种划分其实非常接近机器学习中常见的一种建模方式特征因子分解 / 因子建模（factorization thinking） ——将复杂问题拆解为几个主要因素，再去学习这些因素之间的关系。用户是行为的发起者，物品是被推荐的内容，而场景则描述了交互发生的具体情境。

当这样理解时，推荐问题其实就变成了一个非常清晰的预测问题：在某个环境下，某个用户与某个内容之间产生交互的概率是多少。换句话说，推荐系统本质上是在学习一个函数

f(User, Item, Context) → Interaction

现实世界中的用户行为极其复杂，但通过这样的拆解，系统只需要学习三类因素之间的关系，就可以逐渐逼近真实的行为模式。从这个角度看，这种抽象其实体现了一种很典型的工程智慧：先把现实问题拆解为几个关键因素，再让模型去学习这些因素之间的关系。在继续阅读后面的内容时，这种结构又自然引出了推荐系统的两种不同思路。

一种思路是 判别式推荐，系统面对一个用户和一个候选内容，去判断两者之间是否匹配，本质上就是计算一个匹配分数，即
一种思路是 判别式推荐，系统面对一个用户和一个候选内容，去判断两者之间是否匹配，本质上就是计算一个匹配分数，即

Score = f(User, Item, Context)

判别式推荐的核心：打分函数

另一种思路是 生成式推荐，系统不再逐一评估候选内容，而是根据用户历史行为和当前场景，直接生成用户接下来可能感兴趣的内容序列，例如

[I1, I2, …, Ik] = g(User, Context)

生成式推荐的核心：序列生成

这两种方式在形式上看起来完全不同，但读到这里时我会不自觉地产生一个想法：它们真的必须是互相替代的吗？如果从工程角度来看，似乎也可以把两者结合起来。例如:
在生成式推荐中，模型先根据用户历史行为生成一组候选物品，再利用判别式模型进行打分和排序；或者在生成之前，先用判别式方法缩小候选空间，再让生成模型决定推荐序列。这样一来，生成模型可以负责建模用户行为序列，而判别式模型则提供更加稳定的匹配评分。
从直觉上看，这种组合似乎能够同时利用两种方法的优势。我并不确定这种想法最早来自哪里，是推荐系统研究本身的发展，还是受到了近年来深度学习模型设计思路的影响，但读到这里时，我会隐约感觉到：推荐系统的发展似乎一直在两个问题之间来回摆动——是为所有候选内容计算分数，还是直接生成推荐结果。而未来真正有效的系统，也许并不会完全属于其中某一种，而更可能是在不同阶段将两种思路结合起来。

推荐算法的能力演进

在概念层面上，两种推荐范式都依赖对 User、Item 和 Context 的理解。但如果从模型结构来看，它们的输入实际上并不相同。

在判别式推荐中，模型的输入通常是一个 (User, Item, Context) 三元组。系统需要对每一个候选物品进行评分，从而判断用户是否会对该物品产生兴趣。

而在生成式推荐中，模型的输入往往是 用户历史行为序列和当前场景。模型并不会逐一评估候选物品，而是直接生成用户接下来可能感兴趣的内容。因此在这种范式下，Item 不再是输入，而成为模型的输出。

判别式推荐在回答的是：

“这个用户会喜欢这个物品吗？”

系统面对一个用户和一个候选物品，计算二者之间的匹配程度，然后在候选集合中选择得分最高的内容。

而生成式推荐在回答的是：

“这个用户接下来想看什么？”

系统不再逐一评估候选物品，而是直接根据用户历史行为与当前场景生成一个推荐序列。

表面上看，这两种问题的差异似乎只是形式上的变化，但实际上它会影响整个系统的 架构设计、优化目标以及工程实现方式。判别式推荐天然依赖候选集合，因此更容易形成多阶段的推荐流水线；而生成式推荐则更接近端到端建模，希望通过一个统一模型直接生成推荐结果。

如果把视角再拉长一些，从时间维度来看，推荐算法的发展其实呈现出一条非常清晰的能力演进轨迹。这条轨迹并不是简单的技术替代，而更像是一种能力的逐层叠加。

最早期的推荐系统主要依赖 纯ID的记忆机制。在协同过滤模型中，物品本身并不携带语义信息，它只是一个ID。系统通过统计用户行为之间的共现关系，例如“看过A的人也看了B”，来建立推荐规则。此时的推荐系统更像是一种大规模的行为记忆系统。

随着数据规模的增长以及计算能力的提升，推荐系统逐渐进入 深度学习泛化阶段。神经网络开始被用于学习用户和物品的表示（embedding），通过特征交叉和序列建模，将模型学习到的模式泛化到新的用户–物品组合上。在这个阶段，系统已经不再只是简单记忆行为，而开始具备一定的泛化能力。

再往后，一个新的方向逐渐出现——语义ID（Semantic ID）。当物品不再只是一个无意义的ID，而是被编码为带有语义结构的表示时，系统就开始真正“理解”内容。例如，一个视频可能同时包含主题、类别、创作者等多种信息。当这些信息被编码到统一的表示空间中时，新内容即使还没有积累大量用户行为，也有机会被推荐出来。

而在最近几年，大模型的出现又带来了新的可能性。模型不再只是隐式地计算一个匹配分数，而是能够显式地分析用户意图、推理内容之间的关系，甚至给出推荐理由。从某种意义上说，推荐模型开始从一个 模式匹配器（pattern matcher） 逐渐向 具备推理能力的决策模型 演进。

值得注意的是，这几个阶段并不是简单的线性替代关系。即使在今天的大规模推荐系统中，早期的方法仍然在发挥重要作用。例如：

  • 协同过滤 仍然是召回阶段的重要方法
  • 深度学习模型 支撑着排序阶段的大部分预测任务
  • 语义表示与大模型 则更多出现在前沿的生成式推荐探索中

我的更深层的理解是，推荐系统的发展更像是一种 技术层的不断叠加。新的方法出现后，并不会完全替代旧方法，而是与已有技术共同组成整个系统的能力结构。

读到这里时，我其实产生了一种“原来如此”的感觉。因为在前面的阅读过程中，我已经隐约产生过一个猜想：判别式推荐和生成式推荐或许并不是完全独立的发展方向，而是在推荐系统演进过程中不断交叉出现的两种建模方式。看到这里的历史脉络之后，这种直觉某种程度上得到了印证。

与此同时，我也联想到另一个问题：如果回看早期推荐系统的发展，它们在很多时候其实更像是一种 搜索系统的变式。例如协同过滤，本质上是在用户行为空间中寻找“相似用户”或“相似物品”，然后把这些相似结果返回给当前用户。从形式上看，这和信息检索中的相似度搜索其实非常接近。这点在我查证资料时也有了验证：

  • 《Recommender Systems Handbook》Ricci, Rokach, Shapira:早期推荐系统书籍中将推荐系统放在信息检索背景下讨论，他认为推荐系统核心目标是：“providing suggestions for items to be of use to a user”

所以我能理解的是：
推荐系统可以理解为“没有查询词的搜索”

但是即使是早期，推荐系统和检索也是不一样的：

搜索是用户提出Query，而推荐是系统推断用户兴趣，因此
推荐系统是信息过载环境下的信息过滤 / 信息检索的一种形式
或者说推荐系统产生于信息过载问题
在学术研究中，它常被视为信息过滤（information filtering）或个性化信息检索（personalized information retrieval）的一种形式。

在搜索系统中，用户通过 query 表达需求；
而在推荐系统中，系统需要根据用户历史行为推断这种需求。

某种意义上，早期推荐系统更像是在回答这样一个问题：“在所有内容中，哪些内容和当前用户最相似？” 这和搜索系统中的向量检索、相关性匹配有着明显的相似性。

但随着推荐系统的发展，这个问题逐渐发生了变化。系统不再只是寻找“最相似的内容”，而是开始尝试预测 用户未来可能产生的行为序列。也正是在这个阶段，生成式建模开始逐渐进入推荐系统研究的视野。例如，当用户行为被看作一个序列：

User → Item₁ → Item₂ → Item₃ → …

推荐问题就可以被重新理解为 预测序列中的下一个元素。在这种建模方式下，推荐系统就与自然语言模型产生了某种结构上的相似性：语言模型预测下一个 token，而推荐模型预测下一个用户可能交互的物品。

从这个角度看，推荐算法的发展似乎呈现出两条交织在一起的脉络。一条是 能力演进的时间线：从最早的协同过滤记忆模式，到深度学习的表示学习，再到语义ID和大模型推理能力的引入。另一条则是 建模范式的分支：判别式方法强调对候选物品进行评分，而生成式方法则尝试直接生成推荐序列。

这两条路径并不是互相替代的关系，而更像是在不断交叉演化。技术上，它们形成了两种不同的问题表述方式；但在工业系统中，它们往往被组合在同一个推荐架构中。例如生成模型可以负责提出候选内容，而判别式模型则负责更稳定的匹配评分和排序。因此在现实系统中，我们更容易看到的是一种 生成与评分相结合的结构，而不是完全单一的范式。

如果把推荐系统的发展画成一个二维结构，可以大致理解为下面这样的关系：

工业视角：规模化技术

如何在毫秒级时间内从上亿候选中找到最优结果

在阅读工业级推荐系统这一部分时，我意识到：当推荐问题进入真实互联网环境之后，最大的挑战其实已经不再是算法本身，而是 规模（scale）。

一个典型的平台可能拥有上亿用户和上亿个内容。如果系统试图为每个用户计算他与所有内容之间的匹配程度，那么理论上的计算规模将达到：

1e8 × 1e8

这种级别的计算在实时系统中显然是不可能完成的。

因此工业推荐系统面临的核心矛盾其实非常清晰：

推荐质量 vs 计算成本

或者换一种更直观的说法：

全量搜索 vs 实时响应

如果真的对所有候选内容进行完整评分，推荐结果可能非常准确，但系统的响应时间将无法接受；而如果过度简化计算，又会严重影响推荐质量。因此工业系统必须对计算流程进行一种 结构性的重构。

早期判别式规模化技术的出现

在这里，推荐系统采用了一种非常经典的工程思想：Divide and Conquer（分而治之）。也就是说，不再一次性解决完整问题，而是把推荐过程拆分成多个阶段逐步逼近最终结果。

这就是工业推荐系统中最常见的三阶段流水线：

召回 → 排序 → 重排

如果用非常直观的话来理解这三个阶段，可以把它们看作三个不同层次的问题：

召回：找到 可能相关的内容
排序：找到 最可能喜欢的内容
重排：让推荐 整体体验更舒服

工业级推荐系统的三阶段流水线架构
从目标上看，这三个阶段其实承担着不同的职责：
阶段 目标
召回 覆盖尽可能多的潜在相关内容
排序 精确预测用户点击或互动概率
重排 优化整体体验，例如多样性与新颖性

补充：（重排完全体现了和搜索不一样的特质）
当然也必须提到，谁告诉你百度：搜索要搞成推荐！！！搜索就是搜索，推荐就是推荐啊！！！！
通过这种逐层筛选的方式，推荐系统就可以在 计算效率与推荐效果之间取得一种工程上的平衡。系统不再试图在一开始就解决全部问题，而是先快速缩小候选空间，再逐步使用更复杂的模型进行精细判断。

当然，从结构上看，这种流水线架构其实不仅存在于推荐系统中，在搜索系统、广告系统甚至很多大型信息系统中都能看到类似设计。

所以非常值得学习的是：

  • 1.早期工业界在构建推荐系统时，并不是从零开始设计一整套全新的系统，而是在很大程度上借鉴了信息检索系统中已经被验证有效的架构。这其实体现了一种非常典型的工程智慧——在面对复杂系统时，尽可能利用已有成熟技术，而不是重复造轮子。
  • 2.另外大规模系统往往并不是依赖某一个“超级模型”，而是通过合理的结构设计，把复杂问题拆解成多个可以高效解决的子问题。

当然，这种多阶段流水线结构虽然在工业系统中取得了巨大成功，但它也带来了一个新的问题：整个推荐流程被拆分成多个相互独立的阶段，每个阶段都在优化不同的目标。召回阶段关注相关性覆盖，排序阶段关注点击率预测，而重排阶段又需要考虑多样性和用户体验。这种结构在工程上非常稳定，但也意味着系统整体并不是一个真正的端到端优化过程。

生成式推荐的优势

也正是在这样的背景下，生成式推荐 提出了一种完全不同的思路。既然大模型已经能够在序列空间中生成文本，那么是否也可以直接生成推荐结果，而不再依赖复杂的多阶段流水线？

在生成式推荐中，用户的历史交互序列被看作模型的上下文输入，而推荐问题被转化为一个序列生成任务。模型通过 Transformer 等自回归结构，直接生成用户接下来可能感兴趣的物品序列。换句话说，推荐不再是“从候选集合中挑选内容”，而是“根据用户行为生成内容”。

生成式推荐的端到端架构
从系统结构上看，这种方法将原本的 **召回 → 排序 → 重排** 三个阶段压缩为一个统一的模型。这样做的一个重要优势在于，它可以在同一个模型中同时考虑所有信息，而不需要将问题拆分为多个独立模块。

同时从时代发展角度来看，早期受限于计算资料无法实现大规模的计算，而如今是计算资源大幅提高，我们需要解决高算力难以充分利用问题，自然需要的是更加力大砖飞的方式，但在以后算力逐渐饱和，我们肯定最后还是要回到类似第一个判别式，对于基础算法的优化，和结构工程更合理的搭建的过程上。

因此，生成式推荐的出现并不仅仅是模型能力的提升，更代表了一种新的系统设计思路：从 分阶段流水线架构，逐渐走向 端到端生成式架构。

推荐系统的终极目标

在阅读这一部分时，我也不自觉地把这个问题和之前的思考联系在一起。首先重新回看，我认为这并不是一个一定滑向深渊的选择，推荐系统本身无错，追求更好的效果也没有错。

如果从技术角度来看，推荐系统确实是在努力构建一种生态平衡：既要满足用户体验，又要保证创作者能够获得合理的曝光，同时还要维护平台的长期发展。这像是一个三方共赢的协调系统。

但我也保留的疑问：在真实的平台系统中，推荐系统真的在做“生态平衡器”吗？

从算法目标来看，平台往往会优化一系列非常明确的指标，例如点击率（CTR）、观看时长（Watch Time）、互动率（Engagement）等。这些指标在机器学习中是可以被精确建模和持续优化的。而在大规模数据驱动的系统中，最容易被持续强化的往往正是这些可量化指标。

如果沿着这个逻辑继续思考，一个更极端但也更现实的方向可能是：系统不断学习每一个用户的行为模式，并逐渐为每个用户构建一个高度个性化的信息环境。在这种情况下，推荐系统看起来就不再只是一个宏观生态的平衡器，而更像是一种针对每个用户单独调节的信息“沙漏”——信息从平台的巨大内容池中流入，但最终能够进入用户视野的内容，只会是极少数被算法判定为最符合其偏好的部分。

从纯粹的数据指标来看，这种方式往往能够带来更好的效果：点击率更高、停留时间更长、用户行为更加稳定。但与此同时，它也可能带来另外一种问题：用户的信息环境会越来越个性化，而不同用户之间看到的世界可能会越来越不同。

因此，当推荐系统被放在“用户—创作者—平台”这个生态框架中重新审视时，一个新的问题也逐渐浮现出来：平台究竟是在努力维持一个健康的信息生态，还是在不断优化每个个体的信息匹配效率？

这个问题或许并没有一个简单的答案。但毫无疑问的一件事：推荐系统不仅是一套技术系统，它同时也是一种塑造信息环境的基础设施。因此，理解推荐系统，不仅需要理解模型和算法，也需要理解它在真实世界中的社会影响。

这是问题也正是让我对生成式推荐感兴趣的开始。
