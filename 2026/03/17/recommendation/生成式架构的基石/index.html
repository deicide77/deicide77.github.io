一、写在开头：为什么要学习生成式架构？

在当前人工智能的发展中，生成式模型（Generative Models）已经成为核心方向之一。从大语言模型（LLM）到图像生成模型（如 Stable Diffusion），其底层都依赖于一套关键的生成式架构。

作为一个在本科阶段具备深度学习基础的学习者，在阅读本章内容时，我并不满足于“理解概念”，而是希望做到：

  • 能够从原理层理解 Transformer 架构
  • 能够解释 Attention 的本质作用
  • 能够理解 生成式模型为什么必须使用这些结构
  • 对于 Diffusion 模型，建立从 0 到 1 的认知框架

Transformer架构

笔者前言理解

在阅读本节内容的过程中，结合我已有的深度学习基础，我对 Transformer 架构的优势有以下的认识。

首先，在计算层面，Transformer 将原本依赖时间递归的序列建模问题，转化为基于矩阵运算的并行计算问题。相比 RNN 必须按时间步逐步计算的方式（串行），Transformer 可以在整个序列维度上同时进行计算，从而充分利用 GPU 的并行计算能力。这种“去时序依赖”的计算模式，是其能够在大规模数据场景中取得成功的重要原因。

其次，在建模能力上，Transformer 的核心在于自注意力机制。与传统模型依赖逐步信息传递不同，Attention 允许序列中任意位置之间直接建立联系，从而有效解决了长距离依赖问题。在生成式推荐场景中，这一点尤为关键，因为用户的历史行为往往跨度较长，而模型需要从中捕捉长期兴趣与短期意图。

进一步来看，相比 CNN 等基于局部感受野的模型，Transformer 可以在单层中实现全局信息交互，这大大缩短了信息传播路径，提高了建模效率。且也代表着信息不会衰减，信息损失更少，也就导致了在长序列中，它有着高效的信息保存和信息传递效率。从这一角度来看，Transformer 的优势不仅体现在效果上，更体现在其计算结构上的“全局建模能力”。

最后，Transformer 架构具有良好的可扩展性（Scaling）。其性能可以通过增加层数、扩大模型维度以及引入更多训练数据持续提升，这一特性也直接推动了大规模生成模型（如 GPT 系列）的发展。从更宏观的角度来看，Transformer 的成功不仅是一种模型结构的胜利，更是一种计算范式的转变——即从“基于顺序的建模”转向“基于关系的建模”。

5.2.1.1 自注意力机制

当谈到 Transformer 的核心优势时，Attention 机制是一个绕不开的关键创新，它使模型能够高效建模序列中的依赖关系。

在生成式推荐场景中，模型需要同时建模用户的长期兴趣与短期行为偏好，而 Attention 机制正是实现这种“动态信息选择”的核心工具。

具体而言，注意力机制允许模型在全局范围内动态选择与当前预测最相关的信息，并通过权重分配突出关键位置的影响。这意味着模型不再被限制于固定的顺序处理方式，而是可以根据当前任务需求灵活调整关注重点。

作者最后提到的这句话也是注意力机制问题的最精炼的总结：

给定当前的查询（Query），序列中的哪些部分（Key）最相关，它们的内容（Value）应该以多大的权重被聚合起来？

一些新的感悟：

Transformer 相比于 RNN 的关键变化，并不单是模型结构的不同。
在 RNN 中，信息的传递路径是由模型结构预先定义的，必须按照时间顺序逐步传播；而在 Transformer 中，信息的流动是由 Attention 机制动态决定的，模型可以根据当前任务需要，自主选择最相关的信息来源。

即序列建模从“基于固定结构的信息传递”，转变为“基于内容相似度的信息选择”。这种转变使模型不再依赖预设的传播路径，而是直接在全局范围内建模信息之间的关系，从而显著提升了表达能力和灵活性。

因此，可以将这一变化理解为：模型从“按照既定路径处理信息”，转向“主动选择和组织信息”。这种从“传递”到“选择”的转变，是生成式模型能力跃迁的重要原因之一。

进一步联想到更宏观的模型发展过程，这种从“固定路径的信息传递”到“数据驱动的信息选择”的转变，并不能简单等同于“机器学习”与“深度学习”的划分，但它确实反映了深度学习模型演进中的一个重要趋势。

早期模型更多依赖人为设计的结构来约束信息流动方式，而随着模型能力的提升，尤其是在 Transformer 架构中，信息的组织方式逐渐交由数据本身来决定。

从这个角度来看，模型正在从“依赖结构表达能力”$$，走向“依赖数据学习能力”，而 Attention 机制正是这一转变的典型体现。

首先关于自注意力机制，作者提到QKV矩阵的作用；

  • Key（

  • Value（
    ）    ：表示“序列中每个位置实际包含的内容”。

我基于自己的理解是：

这三个变换并不是为了完成三个不同的任务，而是将同一输入映射到三个不同的表示空间中，从而分别承担不同的角色。其中，Query 表示当前的“查询需求”，Key 表示用于匹配的“索引信息”，而 Value 则表示实际被聚合的“内容信息”。

这种设计使模型能够将“信息选择”和“信息表达”分开表示：通过 Query 与 Key 的相似度计算决定关注哪些位置（即“看谁”），再通过加权 Value 完成信息的聚合（即“拿什么”）。

不进行这种分离，而是直接使用同一表示进行匹配与聚合，模型的表达能力将受到显著限制。

一些思考

基于最近的一些理解，我对于QKV设置的理解还有以下：
首先自注意力机制中体现的“信息选择与聚合”思想（可能专业一点叫做：信息展开（Expansion） + 信息聚合（Aggregation）），并不是孤立存在的，而是在深度学习模型中普遍存在的一种设计模式。

  • 在 Transformer 的 FeedForward 层中，模型通常先将特征从低维映射到更高维空间（如从 d 扩展到 4d），以增强表达能力，然后再压缩回原始维度。这一过程可以理解为在高维空间中对信息进行重新组织与筛选，从而保留更有判别力的特征。
  • 在卷积神经网络（CNN）中，卷积操作用于提取局部特征，而池化（Pooling）则用于保留关键信息，实现信息的压缩与聚合。
  • 在图神经网络（GNN）中，节点表示通过对邻居信息的聚合不断更新，本质上也是一种信息融合过程。

目前我的浅薄的理解是：
许多深度学习模型都遵循这样一种模式：
首先通过高维映射或关系建模扩展信息表示能力，然后通过加权、池化或注意力机制对信息进行选择与聚合。
这种“表达能力扩展 + 关键信息聚合”的组合，使得模型既能够捕捉复杂模式，又能够避免信息冗余，是深度学习模型取得成功的重要原因之一。

在理解多头注意力时，作者的类比很形象：将每一个注意力头看作一个“专家“，不同的头分别从不同角度建模序列关系，例如有的关注语义相似性，有的关注位置关系，还有的关注局部模式。

但进一步思考后，我意识到这种类比可能并不完全准确(也许这也是作者将专家引号的原因)。因为在多头注意力中，所有的注意力头是并行参与计算的，并不会像 Mixture of Experts（MoE）那样通过 gating 机制选择部分“专家”参与推理。

这让我逐渐理解，多头注意力更接近于一种“表示空间的分解”：模型将原始表示投影到多个子空间中，在每个子空间中分别建模不同类型的关系，然后再将这些信息融合起来。而不是像专家模型那样进行“路径选择”。

因此，我目前的理解是：多头注意力可以在直觉上类比为“多个专家协同工作”，但在机制上更接近于“多子空间并行建模”，其核心目的在于提升表示能力，而不是进行稀疏的专家选择。

小思考：
在重新思考 Multi-Head Attention 的设计之后，我意识到，它的价值不仅体现在具体的计算形式上，更体现在其作为一种“通用建模框架”的深远意义。

从结构上看，多头注意力通过多个子空间投影并行建模不同的依赖关系，这种设计具有极强的灵活性。且就目前再看一次该机制，可以发现并非完全高效：不同 head 之间往往存在冗余，部分 head 的贡献有限；同时，其分工是通过训练过程隐式形成的，缺乏明确的结构约束与可解释性。

但正是在这种“不过度约束”的设计中，我逐渐体会到 Multi-Head Attention 的真正魅力。它并不是一个被精细雕刻到最优的结构，而更像是一种具有高度扩展性的通用计算范式。在当时的技术背景下，这种设计以相对简单的形式，实现了对复杂依赖关系的高效建模，并展现出远超时代预期的效果。

换句话说，Multi-Head Attention 的成功，并不在于它解决了一切问题，而在于它提供了一个足够强大且开放的基础，使得后续的优化（如 head pruning、sparse attention、multi-query attention 等）能够不断在其之上演进。

从这个角度看，它既是一个“尚不完美”的结构，也是一个“极具生命力”的起点。这种在简单与强大之间取得平衡的设计，或许正是 Transformer 能够持续发展的核心原因之一。（能体会到所谓的简单的美）

通过位置编码的特点解释，我们能明白，实际上时间序列建模就是需要添加进入时间特征，针对第一个映入时间戳信息，我的理解是：

多粒度时间编码通过将时间拆分为小时、天和星期等多个离散维度，并分别学习其嵌入表示，从而使模型能够捕捉用户行为在不同时间尺度上的周期性模式。

1 day - 2 day
100 day - 101 day
行为的时间尺度影响不一样，所以使用log进行建模

这种方式能够更自然地反映用户兴趣的衰减规律，使模型在短期行为和长期行为之间取得更好的平衡。

而针对这两种方式建模的特点，绝对针对于当前时间节点，而弱关系联系；相对则更偏向强关系信息

5.2.1.3. 两种架构范式

两种架构的出现

首先根据Transformer的架构特点，我去理解为什么主要落在这两个架构上，而不是Encoder-only；

首先即是从第一章学到的，推荐问题主要就是：判别式/生成式；

从传统推荐来说，

给定用户 u 和物品 i → 预测 score(u, i)
这种思想只需要把用户，item做embedding，打分求匹配即可；（判别式）
这种思想只需使用双塔模型/表示学习范式；

但是生成式推荐改变了一件根本性的事情：
问题变成了：

Input: 用户行为序列
Output: 下一个 item（序列生成）

这时对于encoder-only它不会生成，只可以理解，所以它没有自回归机制，没有自然的序列输出结构，它本质上不再符合新的任务范式；

而接下来采用两种范式，也是分别出于两种思想：

一类是以Decoder-only为代表的统一序列建模范式，将用户行为和目标物品拼接为单一序列，通过自回归机制隐式完成理解与生成；
另一类是Encoder-Decoder为代表的条件生成范式，显式地将用户建模与推荐生成解耦，通Cross-Attention在生成过程中动态利用用户信息。

5.2.1.3.1. 两种架构的结构差异

Encoder-Decoder架构

根据这个架构在Huggingface中的解释：
“编码器-解码器模型（例如 T5、BART）：这类模型结合了编码器和解码器两种方法，使用编码器理解输入，解码器生成输出。它们擅长序列到序列的任务，例如翻译、摘要和问答。
”

我认为这里的理解就是，先通过encoder对用户进行建模（全局理解）[作者所说：用户的整体模式和长期偏好]
然后再通过自回归去对结果token进行生成

根据以下的架构去理解：

首先encoder使用双向自注意力处理输入序列，每个位置都可以知道所有未知信息，通过前馈网络增强表达能力，LayerNorm保证训练稳定；
下方输出的上下文：用户行为的整体模式（人物画像？）

而在decoder部分，由于我们最终要实现的是推荐，所以在后半的输出过程中，我们要对未来信息进行掩盖，只能让它基于过去信息对未来信息推测（也就是自回归预测）
注：因果自注意力，实际就是通过mask未来的token，让attention计算时，未来参数为0，不分配注意力，实现只能依赖前t-1个token

最后的cross-attention理解：
由于我们训练encoder就是为了得到用户兴趣，所以在encoder时，我们通过设置：

Query：来自 Decoder
Key/Value：来自 Encoder

即在生成推荐时，查询用户信息

最后得到预测；

注：关于masked attention的理解：

本质不是关闭神经元，而是“禁止某些位置参与注意力计算”；
Mask机制是为了解决Transformer并行计算中违反自回归约束的问题，通过在attention score上添加负无穷实现未来信息屏蔽。

Decoder-Only架构

Decoder-Only将输入和输出视为一个连续的序列，通过统一的因果自注意力机制进行处理。

模型本身是一个自回归过程；Decoder-Only 不再显式区分编码与生成阶段，而是将理解与生成统一到同一过程中。这种设计不仅简化了模型结构，还更适合大规模训练和多任务统一建模。

RecGPT：直接将用户行为作为prompt，利用LLM语义直接生成推荐，可以实现可解释性推荐，对话式推荐
OneRec-V2：从该v1到v2的转变，可以看到相较于D-E结构，D-O结构因为其更统一的架构，和仅需依靠scaling规则进行性能增加，取得了更多的推荐工业界的期待

5.2.1.3.2. 两种架构的优劣权衡

根据架构的分析，而作者的解释，我针对内容，做了自己的简单总结：

Encoder-Decoder 是“结构化建模能力强”的架构

  • 可以对不同类型信息做专门建模
  • 可以实现显式的信息选择
  • 可以更自然处理异构输入

但是由于它太过理想的建模过程：

复杂用户行为 → 分路径建模 → 动态关注 → 生成推荐

以及相对复杂的计算，扩展性差，工程更加复杂，所以目前它更像是一个“惊喜设计”的系统，而不是一个落地的推荐系统结构；

Decoder-Only 是“统一-可扩展”的架构

从一开始，我对于GPT系列全部采用D-O架构，我产生对于D-O是否就是比E-D更强的架构，但是现在我认识到D-O的脱颖而出也是“工业哲学”的展现，更统一、易扩展、已部署的才是学术->工程的真实需求；

  • 参数效率极高
    所有参数都服务于同一个目标；
    scaling 时没有结构浪费；
  • 工程效率极高
    单一attention高度适配GPU/TPU，更容易做优化，符合工程迭代，要求性能提升快的要求
  • 生态优势

一个思考

从架构演化的角度来看，Encoder-Decoder 与 Decoder-Only 的关系，其实类似于计算机网络中 OSI 模型与 TCP/IP 模型的关系。OSI 模型在结构上更加完整和清晰，类似于 Encoder-Decoder 架构对信息处理过程的显式分解；而 TCP/IP 模型则是面向工程实现的简化方案，类似于 Decoder-Only 架构通过统一建模提升了效率与可扩展性。当前工业界选择 Decoder-Only，本质上是一次“工程最优”的选择，而非“表达能力最优”的选择。

从这个角度推测，未来 Transformer 架构的发展，很可能不会停留在纯粹的 Decoder-Only，而是朝向一种“部分结构化 + 部分统一化”的混合范式演进，即在保持统一建模优势的同时，引入一定程度的结构归纳偏置，以提升对复杂信息的表达能力。

与计算机网络不同的是，AI 领域仍处于快速演进阶段，尚未形成像 TCP/IP 那样的长期技术锁定，因此未来更有可能根据实际需求持续重构架构形态，而不是被历史包袱所固化。这也意味着，当前的 Decoder-Only 并不一定是终局，而更可能是一个阶段性的最优解。

5.2.1.3.3. 推荐任务中的架构选择

最后从实际应用中，我们通过两种架构的特点可以很好的理解此处作者需要让我们认识到的：不同需求对应不同架构选择。

从任务类型来看，Encoder-Decoder 能提供结构化的信息处理能力 + 显式的信息选择机制使它在需要显示区分用户理解，输入输出异构的过程更加适合；（需要长期个人画像的优质客户的推荐）
Decoder-Only 更适合序列推荐，用户行为建模；

其次面对大规模数据，D-O架构已被验证（GPT / LLaMA）有更好的参数利用率和Scaling Law的高效扩展能力；

最后从系统与部署角度出发，Encoder-Decoder可以在离线或异步场景下预先计算Encoder的输出并缓存，减少在线推理时的计算量，所以如果存在离线+在线混合系统时，E-D架构有其优势；

而D-O统一架构：更适合端到端优化；

最后作者也提到了，推荐系统中的架构选择，本质上是“表达能力、数据规模与工程效率”之间的权衡，而当前 Decoder-Only 的主流地位，更多源于其在规模化与系统优化上的优势，而非对 Encoder-Decoder 的完全替代。

目前学者们正做的事也可以体现，未来即使只是关于Transformer的架构，也需要更多的探索，去找到更好的架构，因此我更倾向地认为：在 AI 领域，由于技术仍在快速发展，并不存在类似计算机网络那样的路径锁定，因此未来的 Transformer 架构更有可能：向“半结构化 + 半统一化”的混合范式演进即：保留 Decoder-Only 的统一建模能力的同时引入 Encoder-Decoder 的结构归纳偏置；

当然AI的发展太过未知，至少目前针对生成式推荐，我们能认识到的是当前对于D-O架构的学习将是重点，而对新范式的探索，也将会是不断前进的方向。

5.2.2. Diffusion模型

关于Diffusion的学习，由于我没有那么多的基础，我目前学习将集中在基础理解上；

Diffusion最早来源于物理中的扩散过程（如布朗运动），在深度学习中首先在图像生成领域得到广泛应用。（DDPM）

Transformer是基于条件规律的建模，保证了生成的逻辑一致性，但是也有一些局限：
生成速度受限于序列长度，难以并行化；对于某些任务，严格的顺序约束可能并非必需。

与Transformer序列化的生成过程不同；Diffusion更像是“整体修正”，Diffusion的使用是工程上的迭代优化，它是不断去逼近正确的结果；

根据作者的描述：Diffusion模型与Transformer并非对立关系，而是互补的。，在DiT中Transformer作为其去噪网络的骨干；
我认为Diffusion 是一种“生成过程的建模方式”；Diffusion定义的是“怎么生成”，Transformer定义的是“用什么网络来实现这个生成过程”。
所以他们从建模思想上来说是不同的架构（公式），但是可以复用在工程实现的具体结构中；

Diffusion模型的核心机制

我的理解：Diffusion本质是在学习一个“从混乱到有序”的过程：先把真实数据逐步加噪打散成近似随机分布（前向过程），再通过模型学习如何一步步去噪把这种随机状态重新还原成有结构的数据（反向过程）。
因此它不是像Transformer那样逐步生成结果，而是通过多次全局修正不断逼近真实分布；在推荐场景中，可以理解为从一个“随机的用户偏好表示”开始，逐步调整成符合用户历史行为的推荐结果；
而潜在空间扩散模型则是先把高维用户行为压缩成低维语义表示再进行这一去噪过程以提高效率。条件扩散模型则是在去噪过程中持续引入用户历史、上下文等条件信息，使最终生成的结果始终朝着“符合用户偏好”的方向收敛。

相比于Transformer，他的预测将是基于更大范围的个人特征的拟合，而不是逐步的理解；

Diffusion的推荐应用

关于在推荐中的使用：我认为是将“用户或物品的表示学习”转化为一个“从噪声恢复结构”的过程，由于我对Diffusion的过程不了解，所以下列的认识我会更多的尝试更细致的搜索去认识这些过程；

特征增强与表示学习：
Diffusion中的特征增强并不是通过增加数据或筛选特征来实现，而是通过学习一个从噪声到数据的映射函数，使模型能够恢复出更符合真实数据分布的表示，从而隐式地抑制噪声、强化稳定模式，本质上是一种“分布层面的表示优化”。
比如：Diffusion被训练学习：

x₁, x₂, x₃ …（不同噪声版本）

x₁ → x
x₂ → x
x₃ → x

相比正常的x，它对于数据的认识更平滑，学到了领域结构，在推荐系统中类似于（误点急/异常行为（扰动））数据进行训练，可以得到更准确的效果；（重点在模型学到的映射函数更准确、稳定）

用户行为：A → B → C → D（D是噪声）
Diffusion模型：A,B,C,D + noise → 学习恢复 → A,B,C

序列生成与推荐：
Diffusion建模后的序列生成：更适合一段时间的意愿，而不是单点预测，Diffusion的全局去噪得到的是我们在一段时间的序列过程中的整体特征理解；且不同于自回归，可以进行并行去噪，生成过程不受顺序约束；

多模态信息融合

不同模态 → 映射到同一个 latent space → diffusion

Diffusion可以进行全局信息去噪，将文本、图像等多模态信息编码到统一的潜在空间，通过条件扩散生成融合后的推荐结果，实现跨模态的语义对齐。比如是短视频，针对文本内容，视频内容，图像内容可以映射到同一个空间，去学习特征（更统一）

协同过滤与图结构建模：

Diffusion通过去噪、加噪可以得到更好的表示结构，即通过扩散机制增强图结构中的高阶关系建模能力。即相比于普通过滤，在用户-物品交互图的潜在表示上进行扩散，捕捉更复杂的高阶邻居关系和全局协同信号。
