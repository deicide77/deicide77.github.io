从 LLM 到生成式推荐：一个统一建模视角

从Transformer到LLM的演进，本质上并不在于模型结构的根本变化，而在于训练范式的系统化升级。

从结构上看，无论是GPT-2还是GPT-4，其核心仍然是基于自回归机制的Transformer架构，并没有发生本质性改变。然而，模型能力的跃迁，主要来源于训练流程的逐步完善，即：

预训练（Pretraining） → 指令微调（Instruction Tuning） → 偏好对齐（Alignment）

这一三阶段范式，使得模型不仅能够进行语言建模，还能够理解复杂指令、遵循人类意图，并在多任务场景中表现出良好的泛化能力。

因此，现代LLM能力的提升，本质上是“ 训练范式驱动 ”的结果，而非单纯的“模型规模驱动”。

在推荐系统场景中，这一范式可以作为基础起点，但由于推荐任务具有强结构性、个性化和实时性等特点，仍需要在数据建模、目标函数以及推理机制上进行进一步的适配与优化。

LLM建模的三阶段范式

后训练Post-training

InstructGPT论文给出了基于预训练模型之上的后训练流程，展示了使模型从”能生成文本”到”能遵循人类意图”的三个递进步骤；
我的理解即:把“概率模型”变成“行为模型”

  • 预训练：学的是 语言分布 P(text)
  • 后训练：学的是 人类偏好 P(good response | human)

针对InstructGPT的三步，它体现的是一种根据不同的要求选择不同的监督信号进行逐层训练的过程，在Transformer的E-D架构也是
存在类似想法，即将需求按层次设置不同的训练范式；

阶段 监督信号 目的
SFT 正确答案 教模型“像人一样说话”
RM 人类排序 教模型“什么是好回答”
RL 奖励优化 让模型“主动变好”

所以在后训练中，数据标注的好坏，与语言能力，表达能力，认知判断相关；
这就像是一个 “人类认知 × 机器学习优化” 的交叉问题；

所以领域交叉，是AI发展的重要驱动力；

1. 预训练：让模型学会“语言”

预训练阶段本质非常简单：预测下一个 token。

模型不断学习：

在给定上下文的情况下，什么词最有可能出现？

这个目标看起来很朴素，但在大规模数据 + 大模型参数下，会产生一个很关键的现象：

**模型不只是学会语言，还学会了知识、逻辑甚至推理能力
**
也就是所谓的 Scaling Law + Emergent Ability

这一阶段解决的是：
模型有没有“基础表达能力”

2. 指令微调：让模型学会“做任务”

预训练后的模型，现在只是一个“高级自动补全工具”，它能做的也就是被训练的事:预测下一个token。
所以很经典的“幻觉”问题：你问问题，它可能继续编故事，而不是认真回答。这可能是预训练数据不好，也可能是指令微调的效果不行；

第二阶段我们要解决的问题是：

让模型学会理解任务指令并按要求生成输出

做法便是通过构造：$Instruction + Input → Output$，核心是构造高质量的”指令-输入-输出”标注数据。

在这个训练过程中，你的标准数据越优质，越多样，那你的模型能够泛化的指令形式和任务场景就越多，效果就更好。

在指令微调过程中你的训练目标便是，降低条件语言建模损失：

$
[
\mathcal{L}{\text{SFT}} = - \mathbb{E}{(x, y) \sim \mathcal{D}} \sum_{t=1}^{T} \log P_\theta(y_t \mid x, y_{<t})
]
$

$
\begin{aligned}
x &: \text{条件信息（指令+输入）} \
y &= (y_1, \ldots, y_T): \text{目标输出序列} \
y_{<t} &: \text{第 } t \text{ 个 token 之前的输出} \
P_\theta &: \text{模型参数化的条件概率分布}
\end{aligned}
$
关键点在于：损失只在输出token上计算，指令和输入部分不参与梯度更新。

目前关于训练策略：LoRA被广泛采用，是因为它用极小的参数更新，近似实现了全参数微调的效果，但成本大幅降低；此外，LoRA还支持多任务适配，可以在同一基础模型上加载不同的适配器，提高了系统的灵活性和扩展性。
LoRA能成功地核心，即是它定义：任务所需的“行为调整”本身就是低维的
另外基于直觉考虑，预训练模型已经学会了绝大多数“语言能力”，不再需要全参数调整；使大模型微调在实际工程中更具可行性。

3. 偏好对齐：让模型更像“人”

即使经过指令微调，模型仍然会出现：有用性不足 真实性问题 安全性风险
第三阶段：**偏好对齐（RLHF / DPO）**的目的便是：让模型的输出更好地符合人类的价值观和偏好

RLHF 基于人类反馈的强化学习

步骤1：收集人类偏好数据

对于同一个输入提示（prompt）$x$，让模型生成多个输出，然后由人类标注者进行排序，得到偏好对数据：

$$
(x, y^{+}, y^{-})
$$

其中：

  • $y^{+}$：更优输出（chosen）
  • $y^{-}$：较差输出（rejected）

步骤2：训练奖励模型（Reward Model）

训练奖励模型 $R_\phi(x, y)$，使其能够对输出质量进行打分，奖励模型通常基于预训练模型初始化：

$$
\mathcal{L}{\text{RM}} = - \mathbb{E}{(x, y^{+}, y^{-})} \left[ \log \sigma \left( R_\phi(x, y^{+}) - R_\phi(x, y^{-}) \right) \right]
$$

其中：

  • $\sigma(\cdot)$：sigmoid函数
  • $R_\phi$：奖励模型

步骤3：策略优化（PPO）

优化策略模型 $\pi_\theta$，最大化奖励，同时约束其不要偏离参考模型 $\pi_{\text{ref}}$：

$$
\mathcal{L}{\text{PPO}} = \mathbb{E}{x \sim \mathcal{D}, y \sim \pi_\theta} \left[ R_\phi(x, y) - \beta , \mathrm{KL}\left( \pi_\theta(\cdot|x) ,|, \pi_{\text{ref}}(\cdot|x) \right) \right]
$$

其中：

  • $\pi_{\text{ref}}$：参考模型（通常为指令微调后模型）
  • $\beta$：KL约束系数

直接偏好优化（DPO）

DPO跳过奖励模型和强化学习，直接利用偏好数据优化策略：

$$
\mathcal{L}{\text{DPO}} = - \mathbb{E}{(x, y^{+}, y^{-})} \log \sigma \left( \beta \left( \log \pi_\theta(y^{+}|x) - \log \pi_\theta(y^{-}|x) - \log \pi_{\text{ref}}(y^{+}|x) + \log \pi_{\text{ref}}(y^{-}|x) \right) \right)
$$

整体流程总结

  1. 收集人类偏好数据 $(x, y^{+}, y^{-})$
  2. RLHF：
    • 训练奖励模型 $R_\phi$
    • 使用PPO优化策略 $\pi_\theta$
  3. 或使用DPO：
    • 直接优化 $\pi_\theta$，无需奖励模型和强化学习

效果

  • 回答更有用、更简洁
  • 更符合人类意图
  • 减少幻觉与错误
  • 提高安全性与对齐能力

5.3.2. 从LLM到生成式推荐

首先这套自然语言处理领域的技术体系，背后三阶段的本质是：

基础能力 → 任务理解 → 偏好优化

而生成式推荐则是要将LLM的“三阶段能力构建过程”，映射为：从“语言建模” → “用户行为建模” → “个性化决策优化”

思维上来说，这本身和生成式推荐有一定的重合，但是并不完全一致；但是这样的工程思想却是值得针对具体的推荐场景进行适配改造的；

5.3.2.1. 三阶段范式的推荐映射

预训练阶段

推荐系统面对的是：用户行为序列物品 ID 多模态信息；在这样的场景下，要解决的问题是：如何在统一模型中，同时建模：语言语义能力 用户行为模式

因此在生成式推荐中，我们需要精心设计预训练任务和数据配比。确保对于语言数据和行为数据分析的平衡。

个人思考：如果要解决这种问题，我们可能尝试的就是模态对齐，和多任务预训练；

指令微调阶段

LLM学习本身的特质，导致它可以将各种任务统一成一个范式，而对于推荐系统同样面临多样化的任务需求：召回、排序、重排、解释、对话式推荐等。如何将这些推荐任务指令化？

将TokenID翻译成语义表示，即将原本的低信息价值的ID，转变成具有语义信息的Token- ID是当前的连接传统推荐数据和生成式模型的关键桥梁。

偏好对齐阶段

LLM通过人类反馈学习符合价值观和偏好的生成策略。而推荐系统面对的是更广泛的个人，如今的LLM发展由于需要更好的工程落地需求，以及人类范围类有一个大概的共识，所以价值观问题不大，而偏好生成，由于问答是个长时间过程，我们能使用长时间维度的问答，去实现个性化；

但是，推荐系统中，很多个人操作往往是低价值的，即：你的一些搜索不是完全的长时段有效，你的购买行为或者观看行为是复杂的，反馈也是隐式的（点击、观看时长、跳过等），所以推荐场景的偏好对齐将涉及更多的内容过滤、算法优化，以及范式建立；
很好的是，NLP中LLM的成功给推荐系统场景下的使用已经提供了很好的借鉴，相比传统推荐->生成推荐的思考，有更多可以发散思考的空间；

作者提到的便是很好的总结：
如何从隐式反馈中构造有效的偏好信号？如何在多目标优化中找到平衡点？
这些问题使得推荐场景的偏好对齐比LLM更加微妙和富有挑战性。

生成式推荐的核心挑战总结

  • 物品 Token 化问题
    推荐系统中的 item 是 ID（离散符号），而 LLM 只能理解 语义 token

  • 协同信号融合
    LLM擅长内容理解，但推荐本身也有它的优点：用户-物品交互模式（协同过滤）
    如何将这种协同信号分析能力，注入到以语言模型为基础的生成式架构中，这需要精心的设计；

  • 冷启动问题
    新上架的物品、新注册的用户缺乏交互历史，但是LLM本身的语义理解能力，可以快速建立起推荐能力；
    如何将这种优势发挥出来，通过将语义特点从LLM聊天中进行提取，作为新的token价值，或是建立新的架构模式，去让物品信息更好的存放在LLM中，这些都将是推荐场景下可能需要解决的问题。

  • 实时性要求
    在线推荐服务通常要求在数十毫秒内完成推理，生成式模型的自回归特性决定了它需要逐个token生成输出，推理延迟可能达到数百毫秒甚至数秒；如何在保证推荐效果的同时解决效率问题；

综上，我认为生成式推荐的挑战，本质是如何在统一的生成模型中，同时解决表示学习、行为建模、泛化能力和系统效率这四大问题。

要完成生成式推荐，我们需要做的也不是简单的将语言模型应用在推荐任务，而是新的工程构建和模型范式的搭建，尽管十分类似，但忍让需要针对推荐场景的独特性进行深度适配。

结尾

LLM 带来的最大价值，不只是模型能力的提升，更在于提供了一种全新的建模范式：将复杂问题统一转化为序列生成问题。在这一视角下，推荐系统不再被拆分为召回、排序、重排等相互独立的模块，而是可以被整体视为一个条件生成过程，由模型端到端地进行建模与优化。

这种统一建模方式不仅简化了系统架构，还天然支持多任务学习、知识迁移与可解释性建模，标志着推荐系统正在从“工程驱动”走向“模型驱动”，从“规则主导”迈向“学习主导”的范式跃迁。

然而，这种范式转变并非简单替换模型即可实现。正如前文所分析的，生成式推荐需要解决表示、协同、泛化与效率等一系列结构性挑战。因此，真正的关键在于：如何借助LLM的建模能力，对推荐问题进行重新抽象与重构，将原本分散且复杂的任务统一到一个可学习的生成框架之中。

在这一新的问题驱动下，我将在教程引导下进一步深入生成式推荐的核心技术实现。从最基础也是最关键的环节——物品表示与Token化（Item Tokenization）开始，逐步探讨如何将推荐系统中的离散ID、行为序列与多模态信息转化为模型可理解的输入形式，并构建连接传统推荐与生成式模型的桥梁。
