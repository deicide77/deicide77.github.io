从 LLM 到生成式推荐：一个统一建模视角

最近在系统梳理大模型的时候，有一个很清晰的感受：LLM 真正重要的，其实不是 Transformer 本身，而是它背后的“训练范式”。

如果只看模型结构，从 GPT-2 到 GPT-4 并没有本质变化，核心仍然是自回归 Transformer。但能力的飞跃，来自一套逐渐成熟的流程：

预训练 → 指令微调 → 偏好对齐

这三步，本质上是在回答三个问题：

  • 模型如何“会说话”（语言能力）
  • 模型如何“听懂人话”（任务理解）
  • 模型如何“说人话”（符合人类偏好）

1. 预训练：让模型学会“语言”

预训练阶段本质非常简单：预测下一个 token。

模型不断学习：

在给定上下文的情况下，什么词最有可能出现？

这个目标看起来很朴素，但在大规模数据 + 大模型参数下，会产生一个很关键的现象：

模型不只是学会语言，还学会了知识、逻辑甚至推理能力

也就是所谓的 Scaling Law + Emergent Ability

可以简单理解为：

数据 × 参数规模 → 能力“涌现”

这一阶段解决的是：
模型有没有“基础表达能力”

2. 指令微调：让模型学会“做任务”

预训练后的模型，其实只是一个“高级自动补全工具”。

比如你问问题，它可能继续编故事，而不是认真回答。

所以第二阶段要解决的问题是：

如何让模型理解“这是一个任务，而不是续写文本”？

做法是构造：

Instruction + Input → Output

比如：

  • 写总结
  • 做分类
  • 回答问题
  • 多轮对话

关键点在于：

loss 只算在 output 上
模型开始学会“条件生成”

这一阶段带来的变化非常大：

模型从“会说话” → “会干活”

3. 偏好对齐：让模型更像“人”

即使经过 SFT，模型仍然会出现：

  • 啰嗦
  • 跑题
  • 幻觉
  • 不安全内容

原因很简单：

SFT 学的是“人类怎么回答”，不是“什么回答更好”

于是引入第三阶段：偏好对齐（RLHF / DPO）

核心思想：

不再直接教模型答案
而是教模型“哪种答案更好”

流程可以理解为：

  1. 生成多个答案
  2. 人类排序
  3. 训练一个“评分模型”
  4. 再用这个评分去优化模型

最终效果是：

  • 更简洁
  • 更可靠
  • 更符合人类意图

4. 一个关键转变：从“语言模型”到“决策模型”

到这里，其实可以看到一个本质变化：

LLM 不只是生成文本，而是在做“序列决策”

这点非常重要，因为它直接决定了：

LLM 可以迁移到推荐系统

5. LLM → 推荐系统：不是套用，而是重构

很多人第一反应是：

能不能直接用 LLM 做推荐？

答案是：可以，但必须改。

因为推荐系统和 NLP 有本质差异：

（1）输入不再是纯文本

推荐系统面对的是：

  • 用户行为序列
  • 物品 ID
  • 多模态信息

问题来了：

模型怎么理解“item_id = 12345”？

这就引出了核心问题：

物品 Token 化

（2）信号来源变了

NLP 学的是“语义关系”，但推荐更重要的是：

协同过滤信号（谁和谁一起被点击）

这类信息：

无法从文本中直接学到

所以必须解决：

如何把协同信号融入 LLM？

（3）反馈机制不同

LLM 用的是：

  • 人类标注（显式偏好）

推荐系统用的是：

  • 点击 / 停留时间 / 跳过（隐式反馈）

所以问题变成：

如何从行为中构造“偏好数据”？

（4）工程约束更强

推荐系统是在线系统：

  • 延迟要求：几十 ms
  • 而 LLM：自回归生成，天然慢

这带来一个现实问题：

生成式推荐如何落地？

6. 一个统一理解（很重要）

如果把推荐系统重新抽象，其实可以变成：

给定用户历史 → 生成下一步行为序列

也就是说：

推荐问题 = 序列生成问题

这也是为什么：

Transformer + LLM 会成为下一代推荐系统的核心方向

7. 我的理解（这一段你面试可以说）

我自己的一个总结是：

  • 传统推荐：拆成召回 / 排序 / 重排（工程驱动）
  • 生成式推荐：统一建模（模型驱动）

本质变化在于：

从“预测分数” → “生成结果”

这不仅仅是方法升级，更是范式变化。

结尾

LLM 带来的最大价值，不只是模型更强，而是提供了一种新的建模方式：

把复杂问题统一成序列生成问题

而推荐系统，正好是最适合被重构的领域之一。
