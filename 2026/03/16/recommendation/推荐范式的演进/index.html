5.1 <推荐范式的演进 > 笔记

推荐范式的演进：从判别到生成

在阅读这一节时，我有一种“猜测被验证”的感觉。

在前面的学习中，我认识到：
虽然推荐系统经历了从协同过滤到深度学习模型的巨大变化，但其底层问题似乎始终没有改变——系统始终在做判别式推荐；

这其实正是现代推荐系统的核心逻辑。

从最早的协同过滤，到矩阵分解，再到如今复杂的深度神经网络模型，推荐系统的目标始终是学习一个评分函数：

系统首先从海量物品中快速召回一小部分候选，然后通过越来越复杂的模型逐步进行筛选，最终得到推荐列表。

这种架构在工程上非常成功，但随着系统规模不断扩大，一些更深层的问题也逐渐显现出来。在这样的背景下，一种新的思路开始出现：

如果推荐问题本身不是“给候选物品打分”，而是“生成用户接下来可能产生的行为”，会怎么样？

教程在这里提出了一个非常重要的范式转变：

局部打分决策 → 全局概率建模
多阶段级联 → 端到端优化
固定候选集合 → 开放生成空间

从概念上看，我大致能够理解这种变化意味着什么，但具体如何实现这种建模方式，我仍然处在一种“似懂非懂”的状态。这也正是我接下来希望继续学习生成式推荐的原因。

判别式推荐建模方式

虽然在前面博客中，我已经对判别式推荐方法进行了探索，并且教程作者给的公式和解释也很详细，但是为了体现完整性，我们再次梳理一遍，判别式推荐的方法：

推荐系统首先将用户、物品以及上下文特征映射为Embedding向量，然后通过特征交互模块和多层感知机（MLP）进行建模，最终输出一个分数。

这个分数即：

P(click | user, item, context)

也就是用户在当前场景下点击某个物品的概率。

也可以理解为函数：

Score = f(User, Item, Context)

只不过随着模型的发展，研究者不断设计新的结构来提升这个函数的表达能力。

例如，FM、DeepFM、DCN 等模型主要用于学习特征之间的高阶交互关系，而 DIN、SIM 等模型则用于建模用户行为序列，从而刻画用户兴趣的动态变化。

补充：

关于具体判别式做了什么事：
通过视频推荐场景解释
用户是否会点击某个物品，往往不是由单个特征决定的，而是由多个特征组合共同决定的。

例如，一个年轻用户在晚上浏览游戏视频时点击概率可能很高，但同样的用户在凌晨浏览财经视频时点击概率可能会很低。这说明用户行为往往由 特征之间的组合关系 决定，而不仅仅是单个特征的影响。

因此，推荐模型通常需要学习这些特征之间的交互关系。模型首先通过 Embedding 层将离散特征（如用户ID、物品ID、类别等）映射为稠密向量，然后通过特征交互模块来学习不同特征之间的组合模式。例如 FM、DeepFM 等模型可以捕捉特征之间的高阶交互关系，而 DIN 等模型则通过注意力机制关注用户历史行为中与当前物品最相关的部分。

在完成特征交互之后，这些特征表示会被输入到多层感知机（MLP）中进行进一步建模。MLP通过多层非线性变换，将复杂的特征组合映射为一个最终的预测分数，用于表示用户对该物品的兴趣程度。

  • 特征交互：用户是否点击，不是由单个特征决定的，而是由特征之间的组合决定的。

生成式推荐出现契机

  • 参数效率问题：如果用户数1亿，物品数1亿，embedding维度：128，实际参数量为256亿；但一次预测只使用

    1 个 user embedding
    1 个 item embedding
    这导致硬件利用率低，参数利用率低

  • 语义建模缺失:普通生成的ID是随机种子生成ID，模型不知道物品之间关系，传统推荐系统来自同个生态下的产品：Air pods，和iPhone是完全没有语义关系的：1324，2312；这会导致冷启动问题：

    新物品没有用户行为，模型无法推荐
    泛化能力弱，ID没有语义关系

  • 语义建模缺失:该问题最好理解，不做学习解释

从这些问题中可以看到，许多限制其实并不是单个模型结构的问题，而是来自“对候选物品逐个打分”的建模范式本身。正是在这样的背景下，一种新的思路开始出现：如果推荐系统不再枚举候选物品并逐一打分，而是直接学习用户行为序列的生成过程，会不会是一种更自然的建模方式？这也正是生成式推荐研究出现的重要契机。

生成式推荐的核心思想

在这一部分中，对我来说最熟悉却又最陌生的概念其实是 自回归建模（Autoregressive Modeling）。

之所以说熟悉，是因为在深度学习领域中，自回归建模已经非常常见。例如语言模型就是典型的自回归结构：模型通过不断预测“下一个 token”，逐步生成完整的文本序列。

但在推荐系统中重新看到这个概念时，我却产生了一种新的理解。推荐系统可能是最天然适合这种建模方式的场景之一。因为用户行为本身就具有明显的序列结构：用户在某一时间点的行为往往与之前的行为密切相关。例如：

手机 → 手机壳 → 蓝牙耳机 → ?

在这样的序列中，模型真正需要回答的问题其实不是“某个物品是否应该被推荐”，而是：

在已知用户历史行为的条件下，用户接下来最可能与哪个物品发生交互。

从这个角度看，推荐问题与语言模型中的“预测下一个词”其实非常类似。这种自回归建模方式天然地捕捉了用户行为中的序列依赖关系，也使模型能够利用更丰富的上下文信息。

其他的内容通过读第一章，有基本的了解，大概能理解他指向的问题，和解决方案

尤其是生成式模型的诞生，带来的语义ID的低成本标注，是对生成式推荐的产生的极大助力，同时也是很好的解决了判别式推荐的冷启动问题

其次本文也提到了生成式模型和强化学习的关联，我的理解是：
用户行为实际上是一个连续过程。用户在平台上的体验往往由一系列推荐共同决定，而不是由单个物品决定。例如，一个推荐列表是否多样、是否能够持续激发用户兴趣，都可能影响用户是否继续使用平台。
强化学习正是为这种“长期决策问题”设计的。系统可以将用户的行为反馈视为奖励信号，通过不断调整推荐策略来最大化长期收益，例如用户停留时间或长期满意度。

从这个角度看，生成式推荐天然适合与强化学习结合。因为生成式模型本身就是在生成行为序列，而强化学习可以直接对整个推荐序列进行优化，这在传统逐物品打分的判别式框架中是很难实现的。

两种范式的本质区别

结合第一章与第五章目前的初步学习，我对推荐系统两种范式的差异也产生了一些自己的理解。

从判别式推荐到生成式推荐的转变，本质上似乎不仅仅是模型结构的变化，更像是一种建模思想的变化。判别式推荐像是“在一个已经确定的候选集合中进行判断与排序”，而生成式推荐“直接学习用户行为的生成过程”更像是一场个人决策。两种不同建模哲学，实际也是在不同的生产力环境下，不同工程师的思维的碰撞。

对于生成式推荐中，它是对序列数据的自回归学习，但也不局限于序列。基于目前的理解，我也很赞同作者的看法：生成式推荐或许是在尝试模拟一种更接近人类决策的过程。某种程度上，它与当前的大语言模型问答系统有些类似——系统不再只是对既有候选进行打分，而是根据用户历史行为与上下文信息，生成一个更符合用户偏好的推荐结果。从这个角度看，未来的推荐系统也许会逐渐演化为一种“个人推荐助手”。虽然系统服务的是所有用户，但在最终体验上，每个人看到的内容都会呈现出明显不同的个性化倾向。

如果这种方向成立，那么强化学习很可能会成为一个重要的研究方向。因为推荐系统本质上是一个长期决策问题，而强化学习恰恰擅长优化长期收益，例如用户长期满意度、平台生态平衡等目标。

通过以上的学习不禁再次让人惊叹Transfomer的魅力，期待下次Tranfomer的学习,毕竟在这个时代，理解 Transformer 已经不仅仅是理解一个模型结构，更像是在理解一整套新的建模范式。自它产生以来，从自然语言处理到推荐系统，越来越多的问题都开始被统一到序列建模与生成式建模框架之下。
